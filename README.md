# ARENA STRIKE — мобильный онлайн-файтинг 1v1

Дуэль на два устройства в браузере: два бойца, движение вперёд/назад, одна кнопка
атаки с кулдауном 3 секунды, 2 очка здоровья. Побеждает тот, кто первым дважды попал.

- **Backend:** Node.js + [`ws`](https://github.com/websockets/ws), авторитетный сервер
  (движение, радиус удара, кулдаун, здоровье и победа считаются только на сервере).
- **Frontend:** чистый HTML5 Canvas без сборки: процедурные «скелетные» бойцы,
  арена с параллаксом, эффекты, синтезированный звук через Web Audio API.
- Управление: виртуальный джойстик слева + кнопка удара справа; на десктопе `A`/`D` и `Пробел`.

---

## Запуск локально

Нужен Node.js ≥ 18.

```bash
npm install
npm start
```

Сервер поднимется на `http://localhost:3000` и слушает на `0.0.0.0`.

### Игра с двух устройств в одной Wi-Fi сети

1. Запустите сервер на компьютере: `npm start`.
2. В консоли сервер сам напечатает адрес вида `http://192.168.1.42:3000` —
   это ваш локальный IP (если нужно вручную: `ipconfig` на Windows,
   `ifconfig`/`ip addr` на macOS/Linux).
3. Откройте этот адрес в браузере телефона №1 → «Создать комнату» → получите код.
   Тот же адрес и код также показываются на экране ожидания.
4. На телефоне №2 откройте тот же адрес, введите код → «Войти». Бой начнётся автоматически.

> Файрвол компьютера должен разрешать входящие подключения на порт 3000.

## Деплой

Сервер и клиент — один процесс: Node-сервер сам раздаёт статику из `client/`
и принимает WebSocket-подключения на том же порту. Поэтому деплой — это один
web-сервис с командой `npm start`. Порт берётся из переменной окружения `PORT`.

Рекомендуемые платформы (поддерживают постоянные WebSocket-соединения):

| Платформа | Что сделать |
|---|---|
| **Render** | Готовый blueprint — [`render.yaml`](render.yaml): New → Blueprint → выбрать репозиторий. Регион задаётся полем `region` (по умолчанию `frankfurt`). На бесплатном тарифе сервис засыпает — первый заход будет медленным. |
| **Fly.io** | Готовые [`fly.toml`](fly.toml) + [`Dockerfile`](Dockerfile): `fly launch --copy-config`, дальше `fly deploy`. Регион — `primary_region` (по умолчанию `fra`, Франкфурт); от него сильно зависит пинг. |
| **Railway** | New Project → Deploy from GitHub. `PORT` подставится автоматически. |

Пинг виден прямо в игре — цифра по центру вверху HUD (замер WebSocket
туда-обратно каждые 2 секунды): удобно сравнивать хостинги. Если пинг большой,
первым делом проверьте регион сервера — он должен быть ближайшим к игрокам.

⚠️ **Чисто serverless-платформы не подойдут без доработки.** Vercel/Netlify
Functions не держат постоянные WebSocket-соединения и не хранят состояние матча
между вызовами — там понадобится сторонний realtime-слой (Pusher/Ably) или
отдельный WebSocket-сервер. Проще задеплоить всё одним сервисом на Render/Railway/Fly.

### Если фронтенд задеплоен отдельно от сервера

По умолчанию клиент подключается к WebSocket по адресу страницы
(`wss://<хост-страницы>`). Если статика лежит отдельно (например, на GitHub Pages),
укажите адрес игрового сервера одним из способов:

- в `client/index.html`: `window.GAME_WS_URL = 'wss://my-game.onrender.com'`;
- либо параметром URL: `https://мой-фронт/?server=my-game.onrender.com`.

Страница по HTTPS требует `wss://` (клиент подставляет протокол сам).

## Структура репозитория

```
server/index.js    — HTTP-статика + WebSocket: комнаты, симуляция 30 Гц, снапшоты 20 Гц
client/index.html  — экраны (меню/ожидание/итог), HUD, управление
client/style.css   — mobile-first вёрстка, обе ориентации, safe-area
client/js/
  config.js        — определение адреса WS-сервера
  net.js           — обёртка WebSocket + замер пинга
  game.js          — интерполяция снапшотов, предсказание своего движения, камера, рендер
  fighter.js       — процедурный боец: скелетный риг, позы, градиенты
  arena.js         — фон: небо, горы, город, фонари, помост (3 слоя параллакса)
  fx.js            — искры, ударная волна, screen shake, hit-stop
  audio.js         — синтез звуков (Web Audio)
  input.js         — джойстик, кнопка удара с индикатором кулдауна, клавиатура
  ui.js, main.js   — экраны и связка всего
  game-const.js    — константы, общие с сервером
```

## Протокол сообщений (JSON поверх WebSocket)

Клиент → сервер:

| Сообщение | Данные | Смысл |
|---|---|---|
| `create` | — | создать комнату |
| `join` | `code` | войти по коду |
| `move` | `dir: -1\|0\|1` | намерение движения (шлётся при изменении) |
| `attack` | — | попытка удара (сервер проверяет кулдаун и дистанцию) |
| `rematch` | — | готов к реваншу |
| `ping` | `id` | замер задержки |

Сервер → клиенту:

| Сообщение | Данные | Смысл |
|---|---|---|
| `created` / `joined` | `code, idx, lan[]` | вы в комнате (idx — ваш боец) |
| `countdown` | `ms` | старт отсчёта, матч сброшен |
| `state` | `phase, p[2]{x, f, hp, st, cd}` | снапшот 20 Гц: позиция, направление, здоровье, состояние, остаток кулдауна |
| `ev` | `k: attack\|hit\|whiff\|go` | мгновенные события для эффектов/звука |
| `over` | `winner, reason` | конец матча |
| `left` | — | соперник отключился |
| `err` | `msg` | ошибка (нет комнаты и т.п.) |

Сетевое сглаживание: клиент рендерит соперника с задержкой ~110 мс, интерполируя
между двумя снапшотами; собственный боец двигается сразу (предсказание) и мягко
сводится к серверной позиции.

## Графика

Вся графика процедурная (Canvas 2D): персонажи собраны из частей тела с прямой
кинематикой и плавным блендингом поз (стойка, шаг, замах, удар, попадание, нокаут,
победа), отрисовка с учётом `devicePixelRatio`. Готовые растровые спрайты не
используются, но их легко подключить: положите файлы в `client/assets/` и замените
отрисовку в `fighter.js` / `arena.js` на `drawImage`.
